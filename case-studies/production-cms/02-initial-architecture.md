# 02. Initial Architecture

## Introduction

Every production system starts with a simple architecture.

When development of the CMS began, the primary objective was not to support millions of users. The focus was on building a reliable platform that could be developed quickly, deployed easily, and iterated upon as new business requirements emerged.

At this stage, the architecture prioritized simplicity, maintainability, and rapid feature delivery over large-scale optimization.

Looking back, many of these early decisions were exactly what the project required at the time.

---

## Initial Requirements

The platform was designed to solve several core business requirements:

* Create and manage campaigns
* Manage application content
* Configure notifications
* Store user responses
* Generate reports for administrators
* Authenticate administrators
* Manage operational configurations

Although these requirements appear straightforward, they established the foundation for a system that would later process data generated by millions of users.

---

## Architecture Overview

The first version of the platform followed a traditional three-tier architecture.

```text
                  Administrators
                        │
                        ▼
                  CMS (Next.js)
                        │
                        ▼
            Backend (Node.js + Express)
                        │
                        ▼
                    PostgreSQL
                        ▲
                        │
               Consumer Application
                        ▲
                        │
                    End Users
```

The CMS and the consumer application shared the same backend services and database. This simplified development, reduced operational complexity, and allowed features to be delivered quickly during the early stages of the project.

---

## Technology Choices

The initial technology stack was selected with developer productivity and long-term maintainability in mind.

### Frontend

* Next.js
* React
* TypeScript

The CMS was implemented as a web application, providing administrators with a responsive interface for managing platform operations.

### Backend

* Node.js
* Express.js

The backend handled business logic, request validation, authentication, campaign management, and communication with the database.

### Database

* PostgreSQL
* Prisma ORM

PostgreSQL was chosen as the primary relational database, while Prisma simplified database access through a type-safe ORM and improved developer productivity.

### Infrastructure

The initial deployment consisted of a straightforward server setup hosted on DigitalOcean.

Key components included:

* Ubuntu
* Nginx
* PM2
* PostgreSQL

This setup was sufficient for the expected traffic during the early stages of the platform.

---

## Request Flow

The lifecycle of a typical campaign followed a straightforward sequence.

```text
  Administrator
        │
 Creates Campaign
        │
        ▼
   Backend API
        │
        ▼
    PostgreSQL
        │
        ▼
 Campaign Published
        │
        ▼
Consumer Application
        │
        ▼
  User Interaction
        │
        ▼
   Backend API
        │
        ▼
    PostgreSQL
        │
        ▼
CMS Reports & Analytics
```

This architecture allowed administrators to manage campaigns while simultaneously collecting responses from end users through the consumer application.

---

## Database Design

The initial database design focused primarily on supporting application functionality.

Tables were created to represent business entities such as campaigns, users, responses, notifications, configurations, and other operational data.

The schema was intentionally simple, prioritizing rapid development and feature delivery over advanced optimization techniques.

At this stage, database access relied primarily on straightforward CRUD operations through Prisma.

As the amount of stored data increased over time, this simple approach eventually revealed several performance limitations, which are explored in later chapters.

---

## Deployment

The first production deployment followed a relatively simple architecture.

A single DigitalOcean server hosted:

* Backend APIs
* CMS
* PostgreSQL
* Nginx
* PM2 process management

This approach minimized infrastructure complexity and reduced deployment overhead while the platform was still growing.

As traffic increased, the limitations of this deployment strategy gradually became apparent.

---

## Early Assumptions

Many architectural decisions were based on assumptions that were reasonable during the early stages of development.

Some of these assumptions included:

* Traffic would remain relatively moderate.
* Database queries would remain inexpensive.
* Reports could be generated directly from operational data.
* A single server would be sufficient.
* Basic CRUD operations would satisfy most business requirements.
* Infrastructure could remain simple without dedicated optimization.

These assumptions held true for the initial stages of the platform but gradually became less suitable as usage increased.

---

## Looking Back

The initial architecture successfully fulfilled its primary objective: delivering a functional and maintainable platform within a short development cycle.

Rather than introducing unnecessary complexity from the beginning, the system evolved in response to real production requirements.

As the platform grew, increasing traffic, larger datasets, and more demanding administrative workloads exposed new bottlenecks that required architectural improvements.

Those challenges—and the engineering decisions made to address them—form the remainder of this case study.
