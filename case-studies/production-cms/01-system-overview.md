# 01. System Overview

## Introduction

This case study follows the engineering evolution of a production Content Management System (CMS) built to support a large-scale consumer platform.

Over the years, the platform grew to serve **20M+ users**, requiring continuous improvements in architecture, infrastructure, database design, and operational practices.

Like many successful software systems, it did not begin with a sophisticated architecture. The initial focus was delivering features quickly with a simple, maintainable codebase. As traffic increased and business requirements expanded, the system gradually evolved into a production platform capable of handling significantly higher workloads.

This case study documents that engineering journey.

---

## What is the CMS?

The CMS acts as the operational backbone of the platform.

It provides administrators with a centralized interface to manage application content, configurations, campaigns, notifications, user-facing resources, analytics, and various operational workflows.

Although end users never directly interact with the CMS, nearly every action performed by administrators ultimately affects what millions of users experience inside the application.

As the platform expanded, the CMS became increasingly critical to the overall system, making reliability, performance, and scalability essential engineering priorities.

The CMS was not only responsible for managing application content but also for processing operational data generated by millions of end users.

Administrators used the CMS to create and manage campaigns that were delivered to users through the consumer application. As users interacted with these campaigns, their responses and activity were collected by the backend, stored in PostgreSQL, and later retrieved by the CMS to power dashboards, analytics, reports, and data exports.

This continuous cycle of high-volume write operations from users and read-intensive administrative workloads became one of the defining characteristics of the platform, influencing many of the architectural and infrastructure decisions discussed throughout this case study.

## High-Level Architecture

At a high level, requests flow through the following components:

```text
                Administrators
                      │
                      ▼
                 CMS (Next.js)
                      │
                      │
                      ▼
           Backend (Node.js + Express)
             │                    │
             │                    │
             ▼                    ▼
         PostgreSQL       Background Workers
             ▲
             │
             │
        Backend APIs
             ▲
             │
         Consumer App
      (Next.js / Android)
             ▲
             │
         End Users
```

The platform consisted of two primary applications:

- A CMS, used internally by administrators to manage campaigns, content, notifications, and reports.
- A consumer application, used by millions of end users who interacted with the published campaigns and generated operational data.

Both applications communicated with the same backend services, which were responsible for business logic, data processing, and persistence.

As the platform matured, additional layers were introduced to improve performance, reliability, and maintainability. Those architectural changes are discussed throughout this case study.

## Scale

As the platform matured, it grew to support:

* 20M+ users
* Large volumes of daily API requests
* Continuously growing PostgreSQL datasets
* High-frequency read and write operations
* Multiple administrators managing platform content simultaneously
* Report generation and large data exports

Operating at this scale introduced engineering challenges that were not apparent during the early stages of development.

---

## Engineering Challenges

As usage increased, the engineering team encountered challenges such as:

* Increasing database response times
* Expensive SQL queries
* Database connection pool exhaustion
* Rising CPU and memory utilization
* Disk space issues caused by growing logs and datasets
* Worker process management
* API performance bottlenecks
* Infrastructure scaling
* Production monitoring and debugging
* Maintaining high availability during periods of heavy traffic

Each of these challenges became an opportunity to improve the platform's architecture and operational practices.
