# EventLog Timestamp Indexing

## Incident Summary

| Item              | Details                                                                     |
| ----------------- | --------------------------------------------------------------------------- |
| Category          | Database Performance                                                        |
| Severity          | Medium                                                                      |
| Production Impact | Increasing response times for time-based queries and administrative reports |
| Root Cause        | Missing index on a frequently filtered `timestamp` column                   |
| Resolution        | Created a PostgreSQL index concurrently on the `timestamp` column           |
| Downtime          | None                                                                        |

---

## Background

The `EventLog` table was responsible for storing operational events generated by the consumer application. As platform usage increased, the table grew continuously and eventually became one of the largest tables in the production database.

Many administrative features relied on querying recent events for analytics, monitoring, dashboards, and report generation. Most of these queries filtered records based on a time range, such as retrieving events from the previous few hours or days.

During the early stages of the platform, these queries executed quickly because the table contained a relatively small amount of data. As millions of records accumulated, however, the same queries became noticeably slower.

---

## The Problem

One of the most common query patterns looked similar to the following:

```sql
SELECT *
FROM "EventLog"
WHERE "timestamp" >= NOW() - INTERVAL '2 days';
```

Although the query logic itself was simple, PostgreSQL had no efficient way to locate rows matching the requested time range.

As the table continued growing, the database increasingly relied on scanning large portions of the table before identifying the required records.

This gradually affected:

* Administrative dashboards
* Analytics pages
* Report generation
* Operational investigations
* API response times for time-based endpoints

The issue became more apparent as the amount of historical event data increased.

---

## Investigation

The first step was identifying the access pattern rather than immediately attempting to optimize the query.

Several observations became clear:

* Most administrative queries filtered by the `timestamp` column.
* Recent data was accessed far more frequently than older historical records.
* Query performance degraded as the table continued growing.
* The query itself was not unnecessarily complex—the primary issue was locating the relevant rows efficiently.

Rather than redesigning the schema or modifying application logic, the investigation indicated that the database required a more efficient lookup mechanism.

---

## Root Cause

The `timestamp` column, despite being one of the most frequently used filtering columns, did not have its own index.

Without an index, PostgreSQL was forced to examine a significantly larger portion of the table to satisfy each request.

While this behavior was acceptable when the table contained relatively little data, it became increasingly inefficient as millions of additional rows were inserted.

The bottleneck was therefore not the SQL statement itself, but the absence of an appropriate index for the application's primary access pattern.

---

## Choosing the Solution

Since nearly every administrative query filtered records by time, indexing the `timestamp` column was the most appropriate optimization.

Because the database was running in production, maintaining application availability during the operation was an important consideration.

Instead of creating the index using a standard `CREATE INDEX` statement—which can acquire stronger locks and interrupt normal database operations—a concurrent index build was selected.

This approach allowed PostgreSQL to continue serving application traffic while the index was being created.

---

## Implementation

The following statement was executed:

```sql
CREATE INDEX CONCURRENTLY idx_eventlog_timestamp
ON "EventLog" ("timestamp");
```

Using `CONCURRENTLY` ensured that normal read and write operations could continue throughout the indexing process.

Although concurrent index creation generally requires more time than a standard index build, it significantly reduces operational risk for production systems where minimizing downtime is a priority.

For this dataset, the index creation completed in approximately **48 seconds**.

---

## Results

After the index was created:

* Time-based queries completed significantly faster.
* Administrative dashboards became more responsive.
* Report generation improved for queries filtering recent data.
* PostgreSQL no longer needed to scan large portions of the table for common time-range searches.
* The optimization was deployed without requiring application downtime.

The change required no modifications to the application code, demonstrating how an appropriately chosen database index can substantially improve performance with minimal operational impact.

---

## Lessons Learned

Several important engineering lessons emerged from this incident.

### Optimize for Real Access Patterns

Indexes should be designed around how applications actually query data rather than how tables are structured.

---

### Small Datasets Hide Performance Problems

Queries that perform well during development may become inefficient once tables grow to millions of records.

Database performance should always be evaluated at realistic production scale.

---

### Production Safety Matters

For production databases, minimizing operational risk is just as important as improving performance.

Using `CREATE INDEX CONCURRENTLY` allowed the optimization to be deployed without interrupting application availability.

---

### Measure Before Optimizing

The solution was driven by observed production behavior rather than premature optimization.

Understanding the application's actual query patterns led directly to an effective and low-risk improvement.

---

## Takeaway

This incident reinforced an important engineering principle:

> The best database optimizations are often not the most complex—they are the ones that align the database with the application's real-world access patterns.

A single, carefully chosen index eliminated a growing performance bottleneck while requiring minimal changes to the overall system, illustrating how incremental improvements can have a significant impact on production performance.
